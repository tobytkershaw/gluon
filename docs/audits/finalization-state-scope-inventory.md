# Finalization State Scope Inventory

## Purpose

This document inventories mutable state that is not obviously covered by persisted session truth and maps it to an explicit scope.

The goal is to make state ownership auditable before more bugs are found the hard way.

Each entry should answer:

- who owns the state
- what scope it belongs to
- what should reset it
- what should rebuild it
- what risk it creates if left underspecified

## Scope Levels

Use these scopes only:

- `turn`: valid only within one AI/tool execution turn
- `playback`: valid only while playback/runtime scheduling is active
- `project`: valid for one loaded project/session only
- `app`: valid across the app lifetime by design

If a state bucket cannot be assigned clearly, that is itself a design problem.

## Strongest Structural Rule

Any mutable non-session state that can influence AI behavior, user-visible truth, or runtime behavior must have a declared scope, reset trigger, rebuild path, and contract test. Otherwise it is not an optimization or implementation detail; it is an unowned source of drift.

## Current Branch Status

Partially implemented on `codex/hardening-plan`:

- project-scoped AI/UI ephemera now reset on project change
- request staleness moved to an explicit AI turn epoch with invalidation on project changes, provider resets, and project lifecycle actions
- snapshot store reset and compatibility validation are now enforced by contract tests
- playback structural edits now have explicit restart/invalidation coverage

Still intentionally incomplete:

- many human mutation paths still rely on local handler policy rather than a single shared mutation boundary
- some runtime sync caches still need explicit project-boundary reset/rebuild tests
- snapshot compatibility is stronger, but not yet exhaustive for every tool/artifact path

If a state bucket affects:

- AI reasoning
- user-visible diagnostics
- routing/playback behavior
- track/module-targeted operations

and it is not persisted in the session, then it must have:

1. a declared scope
2. a reset trigger
3. a rebuild strategy
4. a contract test

Without those four things, it is a latent drift source.

## Highest-Priority Areas To Harden Proactively

These deserve hardening even where no specific bug has been filed yet.

### AI-side project-scoped ephemera

- patch cache semantics
- any future library/registry/cached inference state
- any “recent context” summaries that survive a turn

Reason:
The audit already proved that long-lived AI-side state leaks across projects unless explicitly reset.

### UI state keyed by entity ids

- selected processor/modulator/module ids
- tracker selections
- deep-view module state
- activity maps
- module-local selection/focus state

Reason:
These tend not to crash immediately, but they silently target missing entities or mislead the user after project or topology changes.

### Runtime sync caches

- dirty-check refs for processor/modulator/source state
- sidechain cache refs
- drum-rack sync assumptions

Reason:
These caches exist to suppress redundant work, but they become correctness bugs if they outlive the state they summarize.

### Artifact stores and references

- audio snapshots
- object URLs
- exported/rendered artifacts referenced by later tools
- any future cached analysis/reference artifacts

Reason:
These are easy to validate only syntactically and easy to misuse semantically.

## Test Recommendations

### 1. Project boundary contract suite

For every `project`-scoped entry above:

- establish initial state in project A
- switch/load/import/create project B
- assert the state is cleared, rebuilt, or replaced according to contract

### 2. Turn boundary contract suite

For every `turn`-scoped entry:

- create the artifact during a turn
- end or supersede the turn
- assert the artifact is unavailable or reset

### 3. Playback boundary contract suite

For every `playback`-scoped entry:

- start playback
- mutate structure or stop/restart
- assert no stale timer/plan/cache can act on missing entities

### 4. Semantic artifact compatibility tests

For any tool that accepts previous artifacts by id:

- reject incompatible scope
- reject incompatible duration when required
- reject incompatible project/session provenance when required
- reject duplicate/self-comparison when required
